# Пульс продаж — интерактивный дашборд

Локальное веб-приложение для управленческого анализа ежедневных продаж из Excel или CSV. Менеджер загружает выгрузку и сразу получает ключевые показатели, динамику и срезы по регионам, магазинам и продуктам.

**Онлайн-версия:** [https://irinazerodev.github.io/sale-dashboard3/](https://irinazerodev.github.io/sale-dashboard3/)

> Данные обрабатываются непосредственно в браузере и не отправляются во внешние сервисы.

## Что получает заказчик

- общий объём продаж;
- количество уникальных транзакций;
- средний чек;
- количество дней с продажами;
- динамику продаж по дням, неделям и месяцам;
- рейтинги регионов, магазинов и продуктов;
- фильтры по периоду, региону, магазину и продукту;
- таблицу исходных операций и экспорт выбранного среза в CSV.

Приложение запускается с демонстрационными данными, поэтому интерфейс можно оценить до получения реальной выгрузки.

## Быстрая проверка

### 1. Склонировать репозиторий

```powershell
git clone https://github.com/IrinaZeroDev/sale-dashboard3.git
cd sale-dashboard3
```

### 2. Создать виртуальное окружение

Windows PowerShell:

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
```

macOS или Linux:

```bash
python3 -m venv .venv
source .venv/bin/activate
```

### 3. Установить зависимости

```powershell
python -m pip install --upgrade pip
python -m pip install -r requirements.txt
```

### 4. Запустить приложение

```powershell
python app.py
```

Открыть в браузере [http://127.0.0.1:8000](http://127.0.0.1:8000), нажать **«Загрузить данные»** и выбрать `test_sales.xlsx`.

Ожидаемый результат: приложение распознаёт 4 380 строк, автоматически выбирает колонки `Дата` и `Продажи`, пересчитывает KPI и активирует все фильтры.

## Требования к системе

- Python 3.12;
- современный браузер: Chrome, Edge, Firefox или Safari;
- свободный локальный порт `8000`;
- интернет после установки зависимостей не требуется.

Проверенные версии Python-библиотек зафиксированы в `requirements.txt`.

## Формат входных данных

Поддерживаются `.xlsx`, `.xlsm` и `.csv` размером до 25 МБ. Анализируется первый лист Excel-книги.

| Бизнес-поле | Обязательное | Примеры заголовков | Назначение |
|---|---:|---|---|
| Дата | Да | `Дата`, `Date`, `Дата продажи` | Фильтрация и временная динамика |
| Продажи | Да | `Продажи`, `Выручка`, `Sales`, `Revenue` | Расчёт объёма продаж |
| Транзакция | Нет | `Транзакция`, `Чек`, `Order_ID` | Количество чеков и средний чек |
| Регион | Нет | `Регион`, `Region` | Региональный срез |
| Магазин | Нет | `Магазин`, `Точка`, `Store` | Сравнение торговых точек |
| Продукт | Нет | `Продукт`, `Товар`, `Product` | Продуктовый срез |

Если названия отличаются, поля можно сопоставить вручную через кнопку **«Настроить колонки»**. При отсутствии идентификатора транзакции каждая строка считается отдельной транзакцией.

Числовые значения могут содержать пробелы, знак валюты, точку или запятую в качестве десятичного разделителя. Даты должны распознаваться Excel либо быть записаны в распространённом формате, например `2026-07-22` или `22.07.2026`.

## Сценарий приёмочной проверки

1. Запустить приложение — должен открыться дашборд с демо-данными.
2. Загрузить `test_sales.xlsx` — в верхней панели должны появиться имя файла и 4 380 строк.
3. Выбрать регион — все KPI, графики и таблица должны обновиться.
4. Изменить период и гранулярность графика — динамика должна перестроиться.
5. Нажать **«Скачать CSV»** — должен сохраниться текущий отфильтрованный срез.
6. Нажать **«Сбросить»** — должны восстановиться полный период и все категории.

## Как рассчитываются KPI

- **Объём продаж** — сумма поля продаж в текущем срезе.
- **Транзакции** — количество уникальных идентификаторов транзакций.
- **Средний чек** — объём продаж, делённый на количество транзакций.
- **Дни с продажами** — количество уникальных дат в текущем срезе.

Все показатели и визуализации реагируют на одни и те же фильтры.

## Архитектура и безопасность

В онлайн-версии встроенная библиотека SheetJS читает первый лист Excel непосредственно в браузере. Агрегации, фильтры и SVG-графики также выполняются на клиентской стороне. Файл не передаётся на сервер и не сохраняется удалённо. Python-сервер оставлен как вариант локального запуска.

Пользовательские значения вставляются в интерфейс как текст, что снижает риск выполнения содержимого из загруженного файла. Онлайн-версия публикуется как статический сайт через GitHub Pages.

## Ограничения прототипа

- в браузер передаются первые 8 000 строк файла;
- используется первый лист Excel-книги;
- для очень больших файлов потребуется серверная агрегация или база данных;
- это самостоятельный веб-дашборд, а не файл Power BI `.pbix`.

## Перспективы развития

- подключение отдельных справочников магазинов и продуктов;
- контроль качества и журнал ошибок загрузки;
- сравнение с предыдущим периодом и планом;
- роли пользователей и публикация во внутренней сети;
- подготовка модели данных и мер DAX для Power BI.
